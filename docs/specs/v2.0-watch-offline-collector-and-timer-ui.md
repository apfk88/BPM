# BPM v2.0.0 Spec: Watch Offline Collector + HealthKit Calorie Import + Timer UI Updates

## 1) Release Metadata
- Marketing version: `2.0.0`
- Build number: increment from `3` to `4`
- Applies to app targets using shared version/build settings.

## 2) Product Goals
- Support workouts when phone is not nearby by using Apple Watch as the collection mechanism.
- Keep iPhone as the main configuration/saving surface.
- Add manual HealthKit import for key calorie profile fields.
- Add a new timer display mode with only a large BPM value.
- Update workout timer end-state stats and button layout.

## 3) Non-Goals (v2.0.0)
- Full standalone watch product parity (history/settings/full editing on watch).
- Cloud sync/account-backed multi-device history merge.
- Replacing existing phone workout flow.

## 4) Watch Scope (Final)
- Watch supports workout capture without phone nearby.
- Watch controls for workout capture:
  - `Start`
  - `Stop/End`
  - `Mark Set`
- Watch is source of truth while active.
- iPhone mirrors/ingests data when reachable.
- Completed watch workouts are retained on watch **until synced** (no TTL).

## 5) Watch-to-Phone Sync Behavior

### 5.1 Transport
- Use `WatchConnectivity`.
- Live state (optional/best-effort): `sendMessage`.
- Durable completed payload transfer: `transferUserInfo` (or equivalent queued transport).

### 5.2 Reliability Contract
- Watch stores completed session payload locally.
- Phone ingests payload idempotently (dedupe by stable session/workout UUID).
- Phone returns ACK.
- Watch deletes local pending payload only after ACK.

### 5.3 Reconnect Experience
- If phone is unavailable, watch capture continues uninterrupted.
- On reconnect, pending payloads auto-transfer in background.
- No manual export step required from user.

## 6) Phone Ingestion and Persistence
- Phone receives watch payload and maps to `WorkoutRecord`.
- `WorkoutStore` save happens automatically.
- `source` should identify watch-origin sessions.
- After local save, phone runs HealthKit sync automatically (same principle as phone workouts).
- If HealthKit sync fails:
  - keep local workout saved
  - preserve failure info for retry/visibility

## 7) HealthKit Calorie Profile Import (iPhone)

### 7.1 UX
- Add button in Calorie Settings: `Import from Apple Health`.
- Trigger is manual only.

### 7.2 Import Modes
- `Fill Empty Only` (default)
- `Overwrite All Imported Fields`

### 7.3 Field Mapping
- `Body Mass` -> weight (`BPM_Calories_WeightKg`)
- `Height` -> height (`BPM_Calories_HeightCm`)
- `Biological Sex` -> sex at birth (`BPM_Calories_SexAtBirth`)
- `Date of Birth` -> age (`BPM_Calories_AgeYears`, computed)
- Optional:
  - `Resting Heart Rate` -> `BPM_Calories_RestHR`
  - `VO2 Max` -> `BPM_Calories_VO2Max`

### 7.4 Result Reporting
- Show per-field status after import:
  - Imported
  - Skipped (existing value preserved)
  - Not found
  - Permission denied/unavailable

## 8) Timer View Mode Changes

### 8.1 Add New View Mode
- Add 4th timer view mode: `bpm`.
- Eye-button cycle order:
  - `table -> stats -> chart -> bpm -> table`

### 8.2 BPM-Only View Rendering
- Show large center BPM only.
- Do not show:
  - substats
  - table
  - charts
  - stat divider rows
- Keep normal top bar and bottom control areas.

## 9) Timer End-State Stats Change
- In workout timer `stats` view, end-state tile currently showing BPM should show HRR instead.
- HRR source:
  - use existing cooldown-based metric already computed in app (`heartRateRecovery` in timer model)
  - do not introduce a new HRR formula in v2.0.0
- Empty HRR display fallback: `---`.

## 10) Timer End-State Button Layout

### 10.1 Beginning State
- No change.

### 10.2 Current/In-Progress State
- No change.

### 10.3 End State (new layout)
- Top row (3 buttons):
  - `Start`
  - `Reset`
  - `Share`
- Bottom row:
  - full-width `Save Workout`

### 10.4 End-State Start Behavior
- `Start` should un-end accidental workout end.
- Behavior: resume timer from end snapshot and continue running.
- Preserve captured sets/samples/session context.

## 11) Data Contracts to Add (Shared)
- `WatchWorkoutStateSnapshot`
- `WatchWorkoutCommand`
- `WatchWorkoutCompletedPayload`
- `WatchSyncAck`

All payloads should be:
- `Codable`
- versioned
- backward-tolerant to future schema additions

## 12) Acceptance Criteria
- Version/build reflected as `2.0.0 (4)`.
- Watch can collect workout while phone is away.
- Reconnect auto-sync imports workout to phone without manual intervention.
- Watch-origin workout persists locally on phone and attempts HealthKit save.
- Calorie Settings can import HealthKit profile fields manually with clear results.
- Timer eye-button cycles through 4 modes including BPM-only.
- Stats end-state shows HRR tile (cooldown-derived) instead of current BPM.
- End-state buttons match:
  - top: Start/Reset/Share
  - bottom: Save Workout

## 13) Test Coverage

### 13.1 Unit
- Watch payload encode/decode and version compatibility.
- Pending queue persistence and ACK-driven deletion.
- Phone ingestion dedupe logic.
- HealthKit import mapping and overwrite mode behavior.
- Timer mode cycle with new `bpm` mode.
- End-state HRR tile value mapping and fallback.

### 13.2 Integration
- Offline watch workout -> reconnect -> auto-ingest on phone.
- Multiple queued workouts sync and dedupe correctly.
- Local save success even when HealthKit write fails.

### 13.3 Manual QA
- Start watch workout, walk away from phone, end workout, reconnect later.
- Confirm workout appears on phone history and sync status behavior is sane.
- Verify button layout and BPM-only mode in portrait and landscape.

## 14) Implementation Notes
- Keep iPhone as primary settings and final save surface.
- Reuse existing HealthKit workout sync service from phone ingestion path.
- Preserve current timer flow for non-watch workouts.
